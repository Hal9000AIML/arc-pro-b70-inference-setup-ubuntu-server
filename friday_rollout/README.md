# Friday 3-Model Rollout (128GB RAM + 4TB SSD install)

Target: 2026-04-10. When the 128GB RAM kit lands on the inference box
(192.168.1.2), this directory has everything needed to go from the current
single-Gemma-4 setup to three co-resident models across the 4× B70 cards.

## What is already verified working (pre-RAM-upgrade)

With the current 16GB RAM + heavy swap configuration, we have
**Gemma 4 26B-A4B running on cards 1,3 (TP=2)** with the workarounds in
`~/start_vllm.sh` (generated by `odin-b70-setup.sh`):

- compute-runtime 25.48.36300.8 + IGC 2.24.8 (do NOT upgrade to 26.09; see below)
- `UR_L0_USE_IMMEDIATE_COMMANDLISTS=0` (routes around libze 25.48 GPF)
- `CCL_TOPO_P2P_ACCESS=0` (forces XCCL allreduce via RAM, P2P hangs TP init)
- `ZE_AFFINITY_MASK=1,3` (cards 0/2 have shown silent SEGV on this rig)
- `--enforce-eager --attention-backend TRITON_ATTN --disable-custom-all-reduce`
- `--skip-mm-profiling --language-model-only` (Gemma 4 is MM, we use text-only)
- `--num-gpu-blocks-override 400` (caps KV cache to avoid XPU allocator
  fragmentation producing 1.9-2.8 GiB contiguous failures)
- `tokenizer.json` merges converted to legacy space-joined string format
  (tokenizers 0.22.2 has multiple parse bugs on the new list format)
- `.pth` hook monkey-patching `Tokenizer.__deepcopy__` to return self
  (works around the pickle roundtrip bug in `copy.deepcopy(kwargs)`)

All of the above are baked into the `vllm-xpu:gemma4-fixed` Docker image
committed by the `odin-b70-setup.sh` post-install hook (section 8b).

## Why we are NOT using compute-runtime 26.09

We upgraded to 26.09.37435.1 / IGC 2.30.1 to fix the 25.48 libze vtable GPF
(`libze_intel_gpu.so.1.14.36300[6e36f5]` during kernel residency setup).

That DID fix the load-time GPF, but 26.09's bundled IGC 2.30.1 has a null
deref during JIT kernel compilation at runtime:

```
VLLM::Worker_TP[...]: segfault at 0 ip ... in libigc.so.2.30.1[+1771512351]
xe 0000:10:00.0: [drm] GT0: Engine reset: engine_class=ccs
```

It crashes on the first actual inference (after model load, after serving
`/v1/models`). Since `libze-intel-gpu1 26.09.37435.1-0` has a strict dependency
`intel-igc-core-2 (>= 2.30.1), intel-igc-core-2 (<< 2.30.1+~)`, we can't
cherry-pick a libze fix without also taking the bad IGC. So we stay on 25.48
and use `UR_L0_USE_IMMEDIATE_COMMANDLISTS=0` to route around the 25.48 GPF.

Re-evaluate when Intel ships IGC 2.30.2+ (watch
https://github.com/intel/intel-graphics-compiler/releases).

## Three-model layout for Friday

Per `project_odin_ram_rollout_plan.md`, but with the hardware-verified card
assignments and the conservative-but-working memory config:

| Tier           | Model                        | Cards      | Port | Script                 |
|----------------|------------------------------|------------|------|------------------------|
| 1 (general)    | Gemma 4 26B-A4B (FP8)        | 1,3 (TP=2) | 8000 | `start_gemma.sh`       |
| 2 (code)       | Qwen3-Coder-30B-A3B (FP8)    | 0  (TP=1)  | 8001 | `start_coder.sh`       |
| 3 (fast/embed) | Qwen3-8B + Qwen3-Embed-4B    | 2  (TP=1)  | 8002/8003 | `start_fast.sh` + `start_embed.sh` |

Notes:
- Tier 1 Gemma is the verified-working TP=2 config on cards 1,3. Do NOT move
  it to 0,1 until we have a chance to rebuild-test cards 0 and 2 on the
  post-RAM-upgrade memory environment.
- Tier 2 Coder-30B-A3B is an MoE with ~3B active, so it fits on a single card
  at FP8. The user specifically asked for the 30B variant for the "biggest
  coder win" (not the 14B).
- Tier 3 Qwen3-8B + Embed-4B share card 2 because neither needs the full 32GB
  alone.
- The pre-upgrade `--num-gpu-blocks-override 400` workaround is KEEP-ON for
  Gemma until we re-benchmark KV cache allocation success on the new 128GB
  RAM / no-swap host. Remove it only after confirming the profile_run OOM is
  gone (it may be triggered by host swap thrashing under the current 16GB).

## Friday execution

1. Install RAM + SSD, power on.
2. `ssh brendanhouck@192.168.1.2`, verify `free -h` shows ~128GB and
   `lsblk` shows the 4TB SSD.
3. Mount SSD at `/mnt/models` (ext4), sync models from current location
   (they should already be there from the 2026-04-08 SSD install).
4. `sudo swapoff -a` + comment out swap in `/etc/fstab` (swap thrashing was
   degrading decode tok/s and likely contributing to the OOM fragmentation).
5. Copy all three `start_*.sh` scripts from this directory to `$HOME`.
6. Copy the three systemd units (`vllm-gemma.service`, `vllm-coder.service`,
   `vllm-fast.service`, `vllm-embed.service`) to `/etc/systemd/system/`.
7. `sudo systemctl daemon-reload && sudo systemctl enable --now vllm-gemma vllm-coder vllm-fast vllm-embed`
8. Wait ~5 minutes for all 4 to health-check.
9. Run `benchmark.sh` to sanity-check tok/s on each port.
10. Update ODIN `config/settings.py` with the three endpoints + restart ODIN.
